AO-WEB-001 [MEDIA] [ESPUBLICO] [WEB] Trafico sospechoso a web server interno

Descripción

Se trata de la regla clonada original de “Suspicious Network Traffic to Internal Web Server” por lo que se debería seguir el mismo procedimiento/investigación que se hacia con ella.

  • Nombre: Trafico sospechoso a web server interno
  • Criticidad: Media
  • Tiempo de Atención: 90 minutos
  • Tecnologías utilizadas: Logs web (sigue el BB de Web servers)

 

MITRE

  • TA0011 - C&C
  • T1102

 

What to do

Una vez salta la alerta debemos fijarnos en la IP origen que está realizando las peticiones, analizarla y poner un contexto sobre la misma (si es maliciosa, si no, si pertenece a una diputación o cliente, etc). Después fijarnos hacia dónde va, por ejemplo, gestiona y ver que tipo de petición es para que haya sido identificada como maliciosa. 

Si tras la investigación se identifica que es un ataque, si es necesario y no se ninguna ip de cliente se puede añadir a la blacklist con el fin de denegar cualquier tipo de ataque procedente de la misma siempre y cuando no haya sido añadida con anterioridad por el sentinel.

Filtrar por la ip realizando las peticiones para verificar si se tratan peticiones puntuales o es posible que este incluso utilizando una herramienta automatizada para testear vulnerabilidades como OWASP ZAP o Nessus

 

Notas de cierre

Puntos mínimos que debe de tener un cierre

  • IP de origen 
  • IP de destino 
  • Si va a gestiona o donde corresponda el ataque
  • Identificar que tipo de ataque se puede llegar a ver en las peticiones
  • Investigación de la IP origen
  • Resumen del análisis

Matriz de escalado

Aunque la matriz de escalado debería ser la misma para todos los casos de uso, puede haber casos que por falta de capacidades, por dependencias de otros servicio o incluso por que así se acuerde para un caso concreto, se debe definir la matriz de escalado.

En horario laboral: Atiende N1 y escala a N2

Fuera del horario Atiende N0 y escala a Guardia

Cierre por defecto

Nota de cierre que se debería de rellenar a la hora de cerrar una alerta:

Falso positivo:

  • Se ha detectado trafico sospechoso hacia la web XXXXX. Al analizar los eventos se observa que el trafico tiene como IP de origen XXXXXX la cual tras ser analizada, se observa que pertenece al dominio XXXXXXXX, y que esta catalogada como maliciosa por [distintos motivos]. En los eventos se observa como realiza un escaneo/peticiones de XXXXXX sin comportamiento malicioso. Se cierra como FP.

Falso positivo con ajuste:

  • No deberia darse el caso.

Positivo:

  • Se ha detectado trafico sospechoso hacia la web XXXXX. Al analizar los eventos se observa que el trafico tiene como IP de origen XXXXXX la cual tras ser analizada, se observa que pertenece al dominio XXXXXXXX, y que esta catalogada como maliciosa por [distintos motivos]. En los eventos se observa como realiza un escaneo/peticiones de XXXXXX con comportamiento malicioso. Ademas, se verifica que la ip esta haciendo multitud de peticiones y escaneos por lo que es posible el uso de herramientas de automatización de pruebas de vulnerabilidades. Se incluye la ip en la blacklist tras verificar que no hay ningun cliente usando la misma.
